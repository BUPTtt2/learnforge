## 二、多Agent架构

### 2.1 架构设计
```
Commander（指挥官）
    ↓ 任务拆分
讲解师（Explainer）
    ↓ 内容生成
审查师（Reviewer）
    ↓ 质量评估
Commander（结果汇总）
```

### 2.2 各Agent职责
| Agent | 职责 | 关键机制 |
|-------|------|----------|
| Commander | 任务规划、进度追踪、结果整合 | 状态持久化、重试控制、降级策略 |
| 讲解师 | RAG检索、内容生成、习题设计 | 工具调用、质量自检、失败重试 |
| 审查师 | 质量评估、问题诊断、决策建议 | 可配置阈值、三种审查策略 |

### 2.3 通信机制
- **直接调用**：Commander同步调用讲解师和审查师
- **状态文件**：JSON文件持久化任务进度
- **结构化输出**：标准化JSON格式便于解析

## 三、RAG检索系统

### 3.1 混合检索实现
1. **向量检索**：SentenceTransformer编码，余弦相似度匹配
2. **关键词匹配**：BM25词频匹配
3. **结果融合**：向量40% + 关键词60%加权
4. **自动降级**：质量<0.5时触发网络搜索

### 3.2 事实一致性保障
- 多维度质量评分（向量30% + 关键词20% + 综合50%）
- 审查师人工把关
- 检索结果重排序机制

### 3.3 幻觉问题解决
- RAG增强：优先从知识库检索
- 外部搜索：本地知识不足时补充
- 提示词约束："不要编造内容"
- 温度调整：降至0.7减少随机性

## 六、面试高频问题

### Q1：为什么升级为多Agent架构？
**A：** 单模型存在质量不稳定、无法处理复杂任务、缺少质量反馈等问题。多Agent实现专业分工（Commander规划、讲解师生成、审查师把关），质量可控，可扩展性强，容错性好。

### Q2：RAG混合检索的权重是如何确定的？
**A：** 通过实际测试确定向量40%、关键词60%。中文技术文档中关键词匹配更准确直接，向量检索捕捉语义关联，该比例平衡了精确性和语义理解。

### Q3：如何处理工具调用失败？
**A：** 1) 重试1-2次（指数退避）；2) 降级策略（RAG→网络搜索→纯模型）；3) 用户友好提示；4) 记录失败日志用于分析。

### Q4：如何保证JSON输出稳定？
**A：** 1) 强格式Prompt；2) few-shot示例；3) JSON Schema验证与修复；4) 降低temperature至0.2-0.3。

### Q5：项目中的自我修正策略？
**A：** 1) 审查师反馈修正；2) 质量阈值自动重试（最多3次）；3) 降级策略；4) 用户反馈收集用于优化。
