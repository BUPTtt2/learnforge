## 一、多Agent框架设计

### 1.1 架构模式：指挥官-专家模式

**设计思想**：
- **Commander（指挥官）**：负责全局协调和流程编排
- **Experts（专家）**：负责具体专业任务

**架构图**：
```
Commander（指挥官）
    │
    ├─── 同步调用 ────► Explainer（讲解师）
    ├─── 同步调用 ────► Reviewer（审查师）
    └─── 同步调用 ────► MemoryManager（记忆管理）
```

### 1.2 Commander 详细设计

**核心能力**：

| 能力 | 实现细节 | 深问 |
|------|---------|------|
| **任务规划** | 根据知识点和掌握度规划章节 | 如何确定章节数量和顺序？ |
| **流程控制** | 编排讲解→审查→更新的闭环 | 如何处理流程中断和恢复？ |
| **状态管理** | JSON文件持久化 | 状态同步的时机如何选择？ |
| **重试机制** | 最多重试2次 | 重试策略如何避免无限循环？ |

**深问解答**：

**Q1：如何确定章节数量和顺序？**
A：章节数量通过模型分析知识点复杂度确定（默认3章），顺序按"基础→进阶→实践"的逻辑排列。

**Q2：如何处理流程中断和恢复？**
A：每个关键步骤后保存状态，恢复时从上次中断处继续执行。

**Q3：重试策略如何避免无限循环？**
A：设置最大重试次数（2次），重试后仍失败则降级返回部分结果。

### 1.3 Explainer 详细设计

**核心能力**：

| 能力 | 实现细节 | 深问 |
|------|---------|------|
| **模式识别** | 简单/深度两种模式 | 模式差异的具体实现？ |
| **内容生成** | 调用大模型生成解释 | 如何保证内容质量？ |
| **练习设计** | 根据知识点设计题目 | 练习题难度如何调整？ |

**深问解答**：

**Q1：模式差异的具体实现？**
A：简单模式使用简洁prompt，限制输出长度；深度模式使用详细prompt，要求包含原理、公式、示例。

**Q2：如何保证内容质量？**
A：生成后经过Reviewer审查，不合格则重试。

**Q3：练习题难度如何调整？**
A：根据掌握度动态调整，掌握度低时题目简单，掌握度高时题目复杂。

### 1.4 Reviewer 详细设计

**核心能力**：

| 能力 | 实现细节 | 深问 |
|------|---------|------|
| **质量评估** | 多维度打分（准确性、完整性、深度） | 评分标准如何确定？ |
| **决策机制** | 通过/重做/部分完成 | 决策阈值如何设定？ |
| **问题定位** | 标记具体问题章节 | 如何精确定位问题？ |

**深问解答**：

**Q1：评分标准如何确定？**
A：使用预设的质量评估prompt，模型输出结构化评分（1-10分）。

**Q2：决策阈值如何设定？**
A：平均分≥8分通过，6-7分部分重做，<6分全部重做。

**Q3：如何精确定位问题？**
A：Reviewer输出问题章节列表，Commander据此标记需要重做的章节。

## 二、RAG检索机制

### 2.1 检索架构

**三层RAG结构**：
```
用户查询
    │
    ├─── 聊天历史RAG → 检索历史对话
    ├─── 外部知识RAG → 检索外部文档
    └─── 学习内容RAG → 检索学习材料
            │
            └─── 结果融合 → 返回给用户
```

### 2.2 检索流程详解

**步骤**：
1. **查询预处理**：对用户查询进行向量化
2. **并行检索**：同时查询三个RAG源
3. **结果融合**：按相关性排序合并结果
4. **后处理**：去除重复，添加来源标识

**深问解答**：

**Q1：向量化模型是什么？**
A：使用Sentence-BERT或OpenAI Embeddings。

**Q2：结果如何排序？**
A：使用余弦相似度排序，不同来源权重相同。

**Q3：如何处理重复结果？**
A：通过内容hash去重，保留相关性最高的。

### 2.3 RAG存储机制

**存储方案**：
- **向量数据库**：Chroma（本地）/ Pinecone（云端）
- **索引结构**：倒排索引 + 向量索引
- **更新策略**：增量更新，定期重建索引

**深问解答**：

**Q1：为什么选择Chroma？**
A：轻量级、本地部署、API友好。

**Q2：索引重建频率？**
A：每天凌晨自动重建，或数据量达到阈值时触发。

**Q3：如何保证检索准确性？**
A：定期评估检索召回率，调整向量化模型和检索参数。

## 三、知识图谱设计

### 3.1 数据结构

**树形结构**：
```
知识图谱
├── 分类节点（14个预定义）
│   └── 知识节点（具体知识点）
└── 关系索引（节点间关联）
```

**节点属性**：
- `name`：节点名称
- `type`：category/knowledge
- `properties`：掌握度、状态、学习历史

### 3.2 索引机制

**设计要点**：
- **节点ID生成**：名称规范化（空格→下划线）
- **位置索引**：`node_index`字典记录节点位置
- **查询复杂度**：O(1)（通过索引直接定位）

**深问解答**：

**Q1：节点ID如何保证唯一？**
A：名称规范化处理，重复添加时更新而非创建。

**Q2：索引如何更新？**
A：添加/删除节点时同步更新索引。

**Q3：为什么不用图数据库？**
A：数据量小（万级以内），JSON足够，便于调试。

### 3.3 掌握度模型

**掌握度计算**：
- 范围：0.0-1.0
- 更新方式：增量更新（通过+0.3，部分完成+0.2）
- 状态切换：≥0.8已掌握，>0学习中，=0未学

**深问解答**：

**Q1：掌握度增量如何确定？**
A：根据学习任务难度和完成质量动态调整。

**Q2：是否考虑遗忘曲线？**
A：当前版本未实现，预留接口便于扩展。

**Q3：掌握度如何影响学习路径？**
A：掌握度高的知识点跳过或快速复习，掌握度低的重点学习。

## 四、记忆系统架构

### 4.1 双层记忆设计

**架构图**：
```
MemoryManager
├── 短期记忆（内存）
│   ├── chat_context：对话上下文
│   ├── rag_results：RAG缓存（5分钟TTL）
│   └── learning_state：当前学习状态
│
└── 长期记忆（持久化）
    └── knowledge_graph：知识图谱（JSON）
```

### 4.2 短期记忆

**特点**：
- **生命周期**：会话级
- **存储位置**：内存
- **清理时机**：会话结束或手动清理

**深问解答**：

**Q1：短期记忆容量限制？**
A：无硬限制，依赖系统内存，建议定期清理。

**Q2：RAG缓存TTL为什么是5分钟？**
A：平衡响应速度和数据新鲜度，可配置。

**Q3：如何处理并发访问？**
A：单进程应用，无需锁机制；多进程需添加锁。

### 4.3 长期记忆

**特点**：
- **生命周期**：持久化
- **存储位置**：磁盘（JSON文件）
- **备份策略**：自动备份到backup目录

**深问解答**：

**Q1：JSON文件大小限制？**
A：理论无限制，建议超过100MB时迁移到数据库。

**Q2：备份频率？**
A：每次写入后自动备份，保留最近10个版本。

**Q3：数据恢复流程？**
A：从backup目录恢复，或使用最新备份。

## 五、缓存系统设计

### 5.1 双层缓存架构

**架构图**：
```
┌─────────────────┐
│ L1 内存缓存     │  100项，1小时TTL
└────────┬────────┘
         │ 未命中
         ▼
┌─────────────────┐
│ L2 磁盘缓存     │  1000项，24小时TTL
└────────┬────────┘
         │ 未命中
         ▼
┌─────────────────┐
│  源数据         │
└─────────────────┘
```

### 5.2 缓存策略

**淘汰策略**：LRU（最近最少使用）
**过期策略**：TTL（时间过期）
**更新策略**：写回（Write-back）

**深问解答**：

**Q1：为什么用LRU？**
A：实现简单，符合访问模式（热门数据经常访问）。

**Q2：TTL如何平衡性能和一致性？**
A：根据数据类型设置不同TTL，RAG结果5分钟，模型响应1小时。

**Q3：缓存穿透如何处理？**
A：使用布隆过滤器过滤不存在的查询。

### 5.3 缓存与记忆的关系

**区别**：
| 维度 | 缓存 | 记忆 |
|------|------|------|
| 目的 | 性能优化 | 功能需求 |
| 数据 | 可丢失 | 不可丢失 |
| 过期 | 必须过期 | 不过期 |

**联系**：
- 缓存可缓存记忆系统的数据
- 记忆系统使用缓存加速访问

## 六、模型管理机制

### 6.1 多模型支持

**模型选择**：
- **本地模型**：Ollama（Qwen3:4b）
- **远程模型**：API（DeepSeek、OpenAI）
- **选择策略**：根据任务类型和性能需求动态选择

**深问解答**：

**Q1：何时选择本地模型？**
A：低延迟要求、隐私敏感场景、无网络环境。

**Q2：何时选择远程模型？**
A：需要更强能力、复杂任务、多模型对比。

**Q3：模型切换如何实现？**
A：通过配置文件或API参数动态切换。

### 6.2 模型调用优化

**优化策略**：
- **请求缓存**：相同请求直接返回缓存结果
- **批量处理**：多个小请求合并处理
- **超时控制**：设置合理超时时间（30秒）

**深问解答**：

**Q1：缓存key如何生成？**
A：使用任务类型+prompt的hash值。

**Q2：超时时间如何确定？**
A：根据模型响应时间统计确定，预留缓冲时间。

**Q3：失败如何处理？**
A：重试1次，失败则降级到其他模型或返回错误。

## 七、模块间通信

### 7.1 通信方式

**同步调用**（主要）：
```python
# Commander 调用 Explainer
result = explainer.generate_explanation(chapter, mode)

# Commander 调用 Reviewer  
result = reviewer.review_all_chapters(knowledge_point, chapters)

# Commander 调用 MemoryManager
memory.update_mastery(knowledge_point, 0.3)
```

**状态共享**：
- **知识图谱JSON**：所有Agent读写
- **Commander状态文件**：Commander读写
- **内存缓存**：所有Agent读写

### 7.2 数据流转

**学习流程**：
```
用户请求 → Commander → MemoryManager(查掌握度) → Explainer(生成) → Reviewer(审查) → MemoryManager(更新) → 返回
```

**对话流程**：
```
用户请求 → RAG检索 → ModelManager(生成) → MemoryManager(存上下文) → 返回
```

### 7.3 一致性保证

**策略**：
- **原子更新**：记忆更新在成功后执行
- **状态校验**：每次操作前校验状态一致性
- **日志记录**：关键操作记录日志，便于追溯
