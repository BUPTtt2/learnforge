# 知识图谱设计详解

---

## 一、整体架构设计

### 1.1 数据结构

**设计思路**：采用树形结构 + 索引加速的混合设计

```
知识图谱
├── 分类节点（category）：根节点（14个预定义分类）
│   └── 知识节点（knowledge）：具体知识点
└── 关系索引（relations）：节点间关联关系
```

### 1.2 核心数据存储

| 字段 | 类型 | 作用 |
|------|------|------|
| `knowledge_graph` | Dict | 树形结构存储，分类 → 知识点 |
| `relations` | Dict | 关系索引，源节点 → [关系列表] |
| `node_index` | Dict | 节点ID → 分类和位置信息 |

---

## 二、节点设计

### 2.1 节点类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `category` | 分类节点 | "人工智能"、"数学" |
| `knowledge` | 知识节点 | "Transformer架构"、"微积分" |

### 2.2 节点属性设计

**通用属性**：
- `status`：学习状态（未学/学习中/已掌握）
- `mastery_level`：掌握度（0.0-1.0）
- `learning_mode`：学习模式
- `history`：学习历史记录
- `created_at`：创建时间

**分类节点特有**：
- `children`：子节点字典

**知识节点特有**：
- `parent`：父分类ID

---

## 三、索引机制

### 3.1 节点ID生成策略

**算法**：名称规范化处理

```
规则：
- 空格 → 下划线
- 斜杠 → 下划线
- 反斜杠 → 下划线
```

**作用**：保证节点ID唯一且可预测

### 3.2 索引结构设计

`node_index` 字典结构：
```python
{
    "node_id": {
        "category": "父分类名称",
        "is_child": True/False  # True=知识节点，False=分类节点
    }
}
```

**检索流程**：
1. 根据名称生成 node_id
2. 查索引 → 得到父分类和类型
3. 直接定位到具体节点 → O(1) 复杂度

### 3.3 分类自动识别

**算法**：前缀匹配 + 包含匹配

```python
# 优先级：前缀匹配 > 包含匹配
for category in categories:
    if name.startswith(category) or category in name:
        return category
```

**默认值**：未识别时默认归为"计算机科学"

## 四、掌握度追踪机制

### 4.1 掌握度模型

| 掌握度区间 | 状态 | 说明 |
|------------|------|------|
| = 0.0 | 未学 | 还没有学习过 |
| (0.0, 0.8) | 学习中 | 正在学习 |
| ≥ 0.8 | 已掌握 | 完全掌握 |

### 4.2 更新策略

**增量更新**：每次学习完成后增加掌握度
- 通过：+0.3
- 部分完成：+0.2

**边界限制**：
```python
new_level = max(0.0, min(1.0, level))  # 保证在 [0,1] 范围内
```

### 4.3 历史记录

每次更新都会记录到 `history`：
```json
{
    "action": "mastery_update",
    "timestamp": "2026-05-12T12:00:00",
    "old_level": 0.5,
    "new_level": 0.8
}
```

## 五、存储方案

### 5.1 持久化方式

**文件格式**：JSON

**存储路径**：`./memory_db/knowledge_graph.json`

**数据结构**：
```json
{
    "knowledge_graph": {},
    "relations": {},
    "node_index": {},
    "saved_at": "2026-05-12T12:00:00"
}
```

### 5.2 选择理由

| 维度 | 说明 |
|------|------|
| **数据量** | 知识点数量有限（通常万级以内） |
| **可解释性** | JSON 可读，便于调试和手动修改 |
| **部署成本** | 无需额外数据库服务 |
| **扩展性** | 结构可演进，预留替换接口 |

## 六、核心功能实现

### 6.1 节点添加流程

```
1. 自动识别分类
2. 检查是否已存在
3. 不存在则创建新节点
4. 更新索引
5. 保存到磁盘
```

### 6.2 节点查询流程

```
1. 生成 node_id
2. 查 node_index → 快速定位
3. 若未找到则遍历搜索（兜底）
```

### 6.3 学习进度统计

**全局统计**：
- 总知识点数
- 已学数量
- 已掌握数量
- 平均掌握度
- 进度率

**分类统计**：
- 每个分类的知识点数量
- 已学数量
- 进度率
- 平均掌握度

## 七、性能优化

### 7.1 时间复杂度

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| 添加节点 | O(1) | 直接插入字典 |
| 查询节点 | O(1) | 索引查找 |
| 更新节点 | O(1) | 直接修改内存 |
| 保存 | O(n) | 序列化写入磁盘 |

### 7.2 优化思路

1. **内存优先**：所有操作先在内存完成，定期持久化
2. **索引加速**：用空间换时间，避免遍历
3. **批量写入**：减少磁盘 IO 次数
4. **懒加载**：只在需要时加载

## 九、面试问题汇总

### 问题1：为什么采用树形结构而不是图结构？

**回答要点**：
- 学习系统中知识分类层次清晰，树形足够表达
- 树形结构简单直观，容易实现和理解
- 查询效率高，不需要复杂的图遍历算法
- 关系通过 `relations` 字段补充，兼顾灵活性

---

### 问题2：节点索引是如何设计的？为什么这样设计？

**回答要点**：
- 用 `node_index` 字典记录每个节点的位置
- 包含父分类信息和类型标记
- 设计目的是将查询从 O(n) 优化到 O(1)
- 空间换时间的经典做法

---

### 问题3：掌握度更新机制是怎样的？

**回答要点**：
- 范围 0.0-1.0，增量更新
- 状态自动切换：≥0.8 已掌握
- 每次更新记录历史，支持追溯
- 预留遗忘曲线接口

---

### 问题4：为什么选择 JSON 文件存储而不是数据库？

**回答要点**：
- 数据量小，JSON 足够
- 可解释性强，便于调试
- 简化部署，减少依赖
- 预留了扩展能力

---

### 问题5：如何保证节点ID的唯一性？

**回答要点**：
- 使用规范化名称作为ID
- 空格、斜杠等特殊字符统一替换
- 重复添加时更新现有节点而非创建

---

### 问题6：知识图谱如何支持自适应学习？

**回答要点**：
- 学习开始时查询掌握度
- 根据掌握度调整学习模式（简单/深入）
- 学习完成后更新掌握度
- 统计数据支持学习进度展示

---

### 问题7：如果数据量变大，如何优化？

**回答要点**：
- 短期：增加缓存层，优化索引
- 中期：迁移到 SQLite/MySQL
- 长期：使用专业图数据库 Neo4j
- 架构上保持接口一致，便于迁移

---

### 问题8：知识图谱和 RAG 是什么关系？

**回答要点**：
- 知识图谱：结构化知识，追踪掌握度
- RAG：非结构化内容，语义检索
- 两者互补：知识图谱提供知识状态，RAG 提供内容
- 通过 MemoryManager 统一管理

---

## 十、总结

本知识图谱设计的核心特点：

1. **简洁高效**：树形结构 + 索引，查询 O(1)
2. **可扩展性**：预留功能和架构扩展接口
3. **易调试**：JSON 存储，可读性好
4. **实用导向**：满足学习系统的核心需求
